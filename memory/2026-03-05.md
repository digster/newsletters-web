# 2026-03-05: Newsletter Archive GitHub Pages Site

## Work Done
- Created the full `newsletters-web/` project from scratch
- Build script (`scripts/build_site.py`): reads from `../newsletters/`, parses YAML front matter from `.md` files, copies `.html` email files, generates `data/index.json` manifest, copies templates to repo root
- Three HTML templates: homepage (card grid), newsletter listing (date-sorted emails), email viewer (iframe with nav)
- CSS with dark mode support, system font stack, responsive design
- Client-side JS with search, keyboard nav (/ to focus, Escape to blur), prev/next email navigation
- GitHub Actions workflow for Pages deployment
- README, ARCHITECTURE, CLAUDE.md, .gitignore

## Stats
- 65 newsletters, 13,641 emails indexed, 13,652 HTML files copied
- 883 MB emails, 3.1 MB manifest
- Build time: ~7 seconds

## Architecture
- Zero dependencies (Python stdlib only)
- Single JSON manifest loaded client-side for all navigation
- Emails rendered in sandboxed iframes to prevent CSS conflicts
- Built files output to repo root (GitHub Pages serves the whole repo)

## Fix: GitHub Pages Deployment
- Added `enablement: true` to `actions/configure-pages@v5` in `.github/workflows/deploy.yml`
- Without this, the action fails with 404 because Pages isn't enabled on the repo

## Verification
- Homepage: 65 newsletter cards with search, email count, date ranges
- Newsletter page: breadcrumb, email count, date-sorted list with subject search
- Email viewer: subject, date, back/prev/next nav, iframe renders original HTML perfectly

# 2026-03-05: Fix Pages Integration Permission Failure

## Work Done
- Investigated GitHub Actions deployment failure caused by `actions/configure-pages@v5` with `enablement: true`
- Removed the `enablement` option from `.github/workflows/deploy.yml` so the workflow no longer attempts to auto-create/enable a Pages site via API
- Kept all other Pages deployment steps unchanged (`checkout`, `configure-pages`, `upload-pages-artifact`, `deploy-pages`)
- Appended session prompts to `PROMPT.md` for traceability

## Why This Fix
- The failing API call requires repository-level permissions that are not available to the workflow integration token in this repository context
- Removing auto-enable avoids the restricted API path while preserving normal deploy behavior once Pages is enabled in repository settings
